Framework DLUnire · Capa de aplicación

DLCore

Kernel funcional que construye sobre el contexto de ejecución provisto por DLRoute. Proporciona ORM, motor de plantillas, autenticación, variables tipadas (.env.type) y el bootstrap de la aplicación mediante Project::run().

DLCore no redefine el entorno de ejecución: lo consume de forma determinista y delega la infraestructura HTTP y de rutas a DLRoute.

  • ORM
  • Plantillas
  • DLAuth
  • .env.type
  • Credenciales
  • Project::run()

Diseño del framework

Arquitectura en capas

Separación intencional entre infraestructura (DLRoute), núcleo de aplicación (DLCore) y persistencia de secretos (DLStorage). Cada capa tiene responsabilidades acotadas.

Contexto de ejecución, document root, peticiones HTTP, enrutamiento y subida de archivos.

Responsabilidades del kernel

  • Consumir el contexto de servidor sin duplicar resolución de rutas base.
  • Orquestar bootstrap, autoload de rutas y ciclo de vida de la aplicación.
  • Exponer APIs estables para datos, vistas, autenticación y configuración.
  • Integrar DLStorage para contenedores .dlstorage y entropía de credenciales.

Licenciamiento

AGPL en la práctica

DLCore es software libre: puede usarlo en proyectos comerciales, cobrar por su producto y escalar su negocio. La AGPL no prohíbe el uso comercial; define cuándo debe compartir el código fuente de su aplicación cuando esta integra el kernel y se ofrece a terceros por red.

En una frase

Puede crear su aplicación con DLUnire. Si terceros usan su app por red y el código de su aplicación no será público bajo AGPL, necesita licencia comercial o publicar fuentes. En desarrollo local, staging, uso interno u open source, no.

composer.json — dependencia mínima

Declare dlunire/dlcore en su proyecto. Composer resolverá dlunire/dlroute y dlunire/dlstorage como dependencias transitivas.

composer.json 
{"require": {"php": ">=8.2","dlunire/dlcore": "^2.1"},"config": {"optimize-autoloader": true}}

Ejecute composer install o composer require dlunire/dlcore para generar vendor/autoload.php.

.env.type — variables con tipo

DLCore valida cada valor al arrancar. Sintaxis: NOMBRE: tipo = valor. El archivo por defecto es .env.type en la raíz del proyecto.

string cadenas entre comillas integer enteros boolean true / false
.env.type 
DL_PRODUCTION: boolean = falseDL_DATABASE_HOST: string = "127.0.0.1"DL_DATABASE_PORT: integer = 3306DL_PREFIX: string = "dl_"MULTITENANT: boolean = false
PHP 
use DLCore\Config\Environment; $env = Environment::get_instance(); if ($env->get_credentials()->is_production()) {// entorno de producción} $db_host = $env->get_env_value('DL_DATABASE_HOST');

Prefijo de tablas — DL_PREFIX

El ORM antepone DL_PREFIX al nombre inferido de la clase. Con DL_PREFIX: string = "dl_", un modelo vacío resuelve la tabla física así:

Products → products → dl_products

final class Products extends Model {} ⇒ dl_products

Archivo .env.type
Sintaxis NOMBRE: tipo = valor
Prefijo ORM dl_

DLCore valida cada valor al arrancar. DL_PREFIX también afecta las tablas inferidas por el ORM — véase modelos.

Capítulo 2 Bootstrap

public/index.php — punto de entrada

El kernel se conecta vía autoload de Composer. Use DIRECTORY_SEPARATOR para rutas portables entre Windows y Unix, sin barras diagonales literales en el código.

Estructura recomendada

mi-proyecto ├── .env.type ├── composer.json ├── public │   └── index.php entrada ├── routes │   └── web.php ├── app │   ├── Controllers/ │   └── Models/ ├── resources │   └── home.template.html └── vendor autoload
public/index.php 
<?php use DLCore\Boot\Project; require_once dirname(__DIR__). DIRECTORY_SEPARATOR . 'vendor'. DIRECTORY_SEPARATOR . 'autoload.php'; Project::run();
Entrada public/index.php
Raíz dirname(__DIR__)
Arranque Project::run()

Resuelve la raíz del proyecto desde public/ y carga vendor/autoload.php con rutas portables vía DIRECTORY_SEPARATOR.

Capítulo 1 Rutas

routes/ — registro HTTP mínimo

Project::run() carga automáticamente los archivos PHP de routes/. Cada llamada a DLRoute registra una ruta; la ejecución ocurre al final del bootstrap.

PHP routes/web.php PHP routes/api.php opcional
routes/web.php 
<?php use DLRoute\Requests\DLRoute; DLRoute::get('/', fn () => ['status' => 'ok',]);
Carpeta routes/
Archivo web.php
Método GET

Project::run() carga cada PHP de routes/ al arrancar. Organice por dominio (web.php, api.php, etc.).

Capítulo 26 Vistas

resources/ — vistas mínimas

Las plantillas viven en resources/ con extensión *.template.html. El nombre de vista usa puntos como separador de carpetas (home → resources/home.template.html).

resources/home.template.html 
<!DOCTYPE html><html lang="es"><head><meta charset="UTF-8"><title>{{ $title }}</title></head><body><h1>{{ $heading }}</h1></body></html>
routes/web.php 
<?php use DLCore\Core\Output\View;use DLRoute\Requests\DLRoute; DLRoute::get('/', fn () => View::get('home', ['title'   => 'Mi app','heading' => 'DLCore operativo',]));
Vista home
Archivo home.template.html
Variables title, heading

Las claves del segundo argumento de View::get() deben coincidir con las variables de la plantilla.

Capítulo 5 Modelos

app/Models/ — modelo mínimo

Un modelo extiende DLCore\Database\Model y mapea una tabla mediante convención de nombres y el prefijo DL_PREFIX definido en .env.type.

app/Models/Products.php 
<?phpnamespace DLUnire\Models; use DLCore\Database\Model; final class Products extends Model {}
routes/web.php 
use DLUnire\Models\Products; DLRoute::get('/api/products', fn () => Products::get());
Prefijo dl_
Modelo Products
Tabla dl_products

Variable DL_PREFIX en .env.type. El ORM antepone el prefijo al nombre inferido de la clase.

Capítulo 3 Controladores

app/Controllers/ — controlador mínimo

Los controladores extienden BaseController, encapsulan la lógica de negocio y devuelven un array que DLRoute serializa según el contexto de la ruta (JSON, vista, etc.).

app/Controllers/HomeController.php 
<?phpnamespace DLUnire\Controllers; use DLCore\Core\BaseController; final class HomeController extends BaseController {public function index(): array {return ['message' => 'DLCore operativo',];}}
routes/web.php 
use DLUnire\Controllers\HomeController;use DLRoute\Requests\DLRoute; DLRoute::get('/', [HomeController::class, 'index']);

Lectura tipada de peticiones, CSRF y validación de entradas.

Capítulo 4 Middleware

Middleware de rutas — DLAuth

Antes de ejecutar el enrutador, logged() y not_logged() envuelven el registro de rutas y sustituyen el handler por HTTP 403 cuando no se cumple la condición de sesión.

routes/web.php 
<?php use DLCore\Auth\DLAuth;use DLRoute\Requests\DLRoute; $auth = DLAuth::get_instance(); $auth->logged(function () {    DLRoute::get('/dashboard', fn () => ['status' => 'privado']);}); $auth->not_logged(function () {    DLRoute::get('/login', fn () => ['status' => 'invitado']);});

Solo afecta rutas registradas dentro de cada bloque en la petición actual.

MULTITENANT — en desarrollo

La variable MULTITENANT en .env.type apunta a un modo SaaS (una base de datos por dominio), pero aún no está terminada. Hoy la entropía de credenciales ya incorpora el host normalizado; el aislamiento multitenant completo depende de DLParse, que sigue en desarrollo.

En despliegues monoinquilino, usa MULTITENANT: boolean = false. Más contexto en el capítulo 13: Credenciales cifradas y DLStorage.
